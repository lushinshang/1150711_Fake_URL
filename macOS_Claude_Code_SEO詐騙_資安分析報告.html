案例來源：Jason Cheng 於 2026/7/10 分享之 macOS + Claude Code SEO 投毒詐騙受害經過（原始貼文詳見 FB文章.html）。本報告以攻防一體視角補充手法定性、框架對應、同類案例查證與防護建議。

1. 手法定性與框架對應

這不是單一漏洞，而是一條完整的社交工程 + 供應鏈信任濫用 kill chain，核心特徵：

  • SEO Poisoning + Malvertising：讓惡意頁面用付費廣告或 SEO 手法排到 Google 搜尋第一名，比官方文件還前面
  • 合法網域信任濫用（Trust Hijacking）：不是偽造網域，而是直接使用 claude.ai 的 Share Link 功能，讓 URL 看起來完全合法（連 VirusTotal 一時都測不出來，因為 domain 本身無罪）
  • ClickFix / Paste-and-Run：誘導使用者手動複製貼上一段 base64 混淆指令到終端機執行，繞過瀏覽器下載掃描與 Gatekeeper
  • macOS Infostealer 酬載：目標是 Keychain、瀏覽器憑證、~/.ssh ~/.aws ~/.kube、Telegram、加密貨幣錢包（Ledger/Trezor 置換）

情報顯示這不是假設情境——2026 年 2-6 月間 Trend Micro、Malwarebytes、Bitdefender、Huntress、CyberSecurityNews 等多家獨立廠商各自報導過幾乎一模一樣的活動群（假冒 Claude Code 安裝頁 + claude.ai/chatgpt.com share link 濫用），案主這次踩到的是一個已經被記錄在案、正在進行中的攻擊活動族群，不是孤例。

2. 初始投放側的同類手法／案例枚舉

💡 觀念澄清：SEO 投毒（Organic）與 惡意廣告（Malvertising）的差別

在許多社群討論中，這兩者常被混為一談。但從技術與防範角度來看，兩者屬於 MITRE ATT&CK 中不同的資源開發戰術（Resource Development）：

  • SEO 投毒（SEO Poisoning, T1608.006）：攻擊者操作搜尋引擎最佳化（SEO）演算法，使其惡意網頁提升至自然搜尋結果（Organic Results）前列。
    • 經典案例：Gootloader 惡意下載器。攻擊者入侵大量合法的 WordPress 網站，並塞入偽裝成法律合約或商業協議範本的網頁與關鍵字。當使用者在搜尋「transition services agreement」等商業範本時，被高排名的惡意論壇網頁吸引，下載了打包成 ZIP 的惡意 JavaScript 腳本，進而引發勒索軟體或 Cobalt Strike 感染鏈。
  • 惡意廣告（Malvertising, T1583.008）：攻擊者直接向廣告平台付費購買廣告（Paid Ads），將連結置於搜尋結果最上方的「廣告 option」或「贊助商」版位。
    • 經典案例：KeePass / Rufus 偽冒下載廣告。攻擊者購買關鍵字廣告（如 "KeePass download"），排在官方網站上方。使用者點擊後會進入 1:1 克隆的偽造官網，下載被植入竊密軟體（如 RedLine, Vidar Stealer）的安裝檔。為了規避 Google 廣告平台的審查，攻擊者使用了「Cloaking（偽裝）」重導向技術，使審查爬蟲看到乾淨的頁面，一般使用者則被引導至惡意下載頁面。

⚠️ 403 代表該 URL 標題經 WebSearch 交叉確認存在，但 WebFetch 遭來源網站擋下（常見反爬蟲機制），未能逐字驗證內文，建議自行瀏覽器開啟核對。其餘連結皆已由 subagent 實際 WebFetch 開啟驗證。

4. 對 vibe coding 使用者的防範柵欄

觀念之外，可以現在就建的技術護欄：

(1) 執行環境隔離

  • 永遠不要在裝有正式憑證的工作機上執行不明來源的一鍵指令——用一台乾淨的 VM/Sandbox 機器或 disposable container 先跑
  • Claude Code 官方提供 sandbox 機制：檔案系統隔離 + 網路 egress 白名單，可降低 84% 權限詢問（Anthropic Engineering）
  • 用 devcontainer 開發時，不要把 ~/.ssh 等 host 端密鑰掛進去，並用 init-firewall.sh 限制 egress（Claude Code Devcontainer 文件）

(2) 指令執行前的強制檢查習慣

  • 任何 curl | bash、curl | sh 型態的一鍵指令，先 curl 存檔看內容，不要直接 pipe 執行
  • 看到 base64、hex 編碼的網址或 payload，先解碼再判斷（echo xxx | base64 -d）
  • 貼上執行前先確認：這段指令是官方文件複製過去的，還是別人分享頁面上臨時出現的——本案的關鍵破口就是「share link 網域是真的，但指令內容是攻擊者塞的」

(3) 憑證與密鑰隔離

  • macOS Keychain、瀏覽器記住密碼盡量少用（案主本身這麼做，才讓損害有限）
  • ~/.ssh、~/.aws、~/.kube 等機敏配置檔，若非必要不要放在會被一般 AI agent / 腳本掃到的路徑，可考慮搬到需要額外解鎖的位置（如 1Password SSH agent、hardware key）

(4) AI Agent 層級的權限最小化

  • OWASP LLM Top 10 的 LLM06 Excessive Agency、OWASP Agentic Top 10 的 ASI03 身分與權限濫用都直接對應「AI agent 被誘導執行超出預期範圍的動作」——使用 mcp_graylog、mcp_pve、mcp_wazuh、mcp_zimbra 等多 MCP 工具鏈時，建議逐一檢視每個 MCP server 的權限範圍是否最小化（OWASP Agentic Top 10 2026）
  • 案主用 Wazuh Agent 事後才發現「這台主機還沒裝」——新機到貨當下第一件事應是先裝好 EDR/Agent，再裝其他工具，這是本案唯一真正的流程漏洞

(5) 網路層防護

  • 貼文中提到 Avast Free for Mac 已能攔截第一、二階段惡意 script 下載——EDR/AV 不是萬靈丹但值得裝
  • DNS 層或防火牆層可考慮擋掉常見 infostealer C2 的已知惡意網域清單（配合既有的 opnsense/zenarmor/adguard）
